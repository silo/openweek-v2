# Data Model

> **Status (re-derived from [`/design`](../design/) `Openweek v2 (paper)`):** the recurrence/subtask columns
> are **active** (self-FKs on `recurrenceId`/`parentId`); `lists.color`, `tasks.startTime` (`'HH:mm'` label),
> `tasks.linkedEventId` (convert-to-task link), a partial unique `(recurrence_id, date)` index, and the sync
> tables (`connected_accounts`/`external_calendars`/`synced_events`) all exist. User appearance fields are
> `fontStyle`/`accentColor`/`tagStyle`/`showCalendarEvents`. All are listed in the tables below. See
> [decisions.md](./decisions.md) D15–D19 and [design.md](./design.md).


PostgreSQL via Drizzle ORM. All **app** tables use **uuid v7** primary keys (time-sortable, double as the
ordering tiebreaker). Better Auth tables keep their native `text` ids; app foreign keys to users are `text`.

## Ownership

- **Better Auth owns** `user`, `session`, `account`, `verification`. The admin plugin adds
  `role` / `banned` / `banReason` / `banExpires` to `user` and `impersonatedBy` to `session`. These tables are
  **generated by `@better-auth/cli generate`** and committed; `drizzle-kit` owns the migrations.
- **App owns** `boards`, `lists`, `tasks`, and the sync tables `connected_accounts`, `external_calendars`,
  `synced_events`.

## Why "boards" not "calendars"

The user's planner container was originally called `calendars` — but the app also mirrors **external**
Google/CalDAV/iCal calendars. Two "calendar" concepts in one codebase guarantees confusion in code, routes,
and UI copy, and it's expensive to rename after migrations and URLs harden. So the planner container is
**`boards`**; the word "calendar" is reserved for external sources.

**Constraints (first migration):**
- `CHECK ((date IS NOT NULL) <> (list_id IS NOT NULL))` — the exactly-one-of invariant, enforced in the DB so
  no code path (including future import/sync) can violate it.
- Composite FK `(list_id, board_id) → lists(id, board_id)` — a task's list must belong to the task's board.
- Indexes `(board_id, date, position)` and `(list_id, position)` — the two ordering scopes.

## Ordering

`position` is a **fractional index** (`fractional-indexing-jittered`), meaningful only **within a single
day-column or single list**. To reorder/insert, compute `generateKeyBetween(prevKey, nextKey)` from the
neighbors at the drop site (`null` at the ends). Jitter avoids identical colliding keys when two devices
insert at the same slot.

**Always `ORDER BY (position, id)`** — never `position` alone. The uuid v7 `id` is a deterministic tiebreaker
on the rare jitter collision. A server-side rebalance routine handles the (rare) deep-nesting/exhaustion case.

## Auto-rollover

Unfinished past tasks roll to "today". **Opt-in** (`user.rolloverEnabled`, default false); a per-user toggle
ships regardless.

- **Trigger:** an explicit authenticated `POST /api/tasks/rollover` called on app load/focus — **never** as a
  side-effect of the SSR grid GET (GET-with-writes is a footgun and double-fires across SSR + hydration).
- **Idempotent:** guard with `user.lastRolloverDate` (and a transaction / pg advisory lock) so it runs
  at-most-once per user per **local** day. The UPDATE is `WHERE date < today AND done = false`.
- **Mutates `date` to today** (matches teuxdeux/tweek mental model) but records `rolledOverFrom` so it's
  reversible and preserves history.
- **Skips recurring instances** (`recurrenceId IS NULL`) so rolling one doesn't collide with the next
  regenerated occurrence.

## Recurrence

A task can repeat (`↻ wkdays|weekly|monthly|daily` in the design). Shape (Google/Asana pattern, see
[decisions.md](./decisions.md) D17): the row carrying the `recurrenceRule` (RFC-5545 RRULE) **is the first
occurrence** — no invisible generator. Later **instances are real task rows** (so they stay
draggable/checkable/taggable) linked by `recurrenceId`, **materialized per visible week** and kept idempotent by
a partial unique `(recurrence_id, date)` index; per-instance overrides/exceptions use the RECURRENCE-ID pattern.
Using **one RRULE representation across both `tasks` and `synced_events`** keeps a future two-way calendar sync a
field-to-field translation, not a model translation. Rollover skips templates and instances.

## Subtasks

`parentId` (depth 1) holds checklist children that share the parent's day/list scope — the design shows progress
as `☑ {done}/{total}`. The top level filters `parentId IS NULL`; rollover skips subtasks.

## Entity sketch

```
user ──< boards ──< lists ──< tasks (date XOR list)
  │                      ▲         │ (self-FK: recurrenceId, parentId)
  │        composite FK ─┘         │ linkedEventId ─────────────┐
  └──< connected_accounts ──< external_calendars ──< synced_events
                                      │                   ▲
                              display-mapped to a board ──┘ (external_calendars.boardId)
```

A task's `linkedEventId → synced_events.id` is the "kept linked" relation from convert-event-to-task; it is
nullable and `ON DELETE SET NULL` (deleting a mirrored event must not delete the user's task).
