# Architecture

How Openweek is structured: the layers, the folder layout, the data flow, and the rules that keep server-only
code out of the client bundle.

## Layers

```
┌────────────────────────────────────────────────────────────────┐
│  Browser (Vue 3.5 / Nuxt 4 client)                             │
│   pages → components/week/* (custom grid)                      │
│   composables/useTaskBoard (Pragmatic DnD)                     │
│   stores/board.ts (Pinia: normalized cache + optimistic writes)│
└───────────────▲───────────────────────────┬───────────────────┘
                │ typed $fetch / useFetch    │ optimistic update + rollback
                │ (validated by shared Zod)  ▼
┌───────────────┴────────────────────────────────────────────────┐
│  Nitro server (server/)                                        │
│   api/auth/[...all]  → Better Auth                             │
│   api/boards|lists|tasks  → CRUD, session-guarded, Zod-checked │
│   api/sync/*  → connect / list / sync-now                      │
│   tasks/sync.ts  → scheduled poll of connected accounts        │
│   utils/{auth,crypto,runtime-config}                           │
│   db/  → Drizzle + pg Pool                                     │
└───────────────▲───────────────────────────┬───────────────────┘
                │                             │
        ┌───────┴────────┐          ┌─────────▼──────────┐
        │  PostgreSQL    │          │ Google / CalDAV /  │
        │  (Drizzle)     │          │ iCal (read-only)   │
        └────────────────┘          └────────────────────┘
```

## The shared-schema rule (prevents client-bundle leaks)

Nuxt 4's `shared/` is for code imported by **both** the Vue client and the Nitro server. The trap: Drizzle
table definitions import `drizzle-orm` runtime and often sit next to the DB client — if a client component
imports a schema file that transitively pulls those in, the DB layer leaks into the browser bundle.

Rule:
- **Drizzle table definitions live in `server/db/schema/` only.** Never import them from `app/`.
- **`shared/schemas/` holds plain Zod DTOs** (request/response shapes). They may be authored by hand or
  derived from drizzle-zod and re-exported as plain Zod objects — but the file the client imports must not
  drag a live `drizzle-orm`/DB module into the client chunk.
- Import shared schemas via the `#shared` alias (e.g. `import { createTaskInput } from '#shared/schemas/task'`).
  Only `shared/utils` and `shared/types` auto-import; `shared/schemas` is imported explicitly (clearer for
  validation).
- Keep DB client, token encryption, and Better Auth secrets in `server/` exclusively.
- Verify with Nuxt 4.4's import-protection trace + a CI check that no DB/table module enters the client chunk.

## Data flow: a task mutation

1. User checks a task / drags it / edits the title.
2. The component calls a **store action** (`board.ts`) — never `$fetch` directly.
3. The store applies the change **optimistically** to its normalized cache (instant UI) and fires the typed
   `$fetch` PATCH/POST, with the body shaped by a `#shared` Zod DTO.
4. The Nitro route re-validates the same Zod DTO, checks the session, writes via Drizzle, returns the row.
5. On success the store reconciles; on error it **rolls back** and surfaces a toast.

Every write — create, edit, toggle-done, color-tag, drag-persist, rollover — goes through this one funnel.
That is what makes a future PWA/offline mode additive rather than a rewrite. `fractional-indexing` already
gives order-merge tolerance for the offline case.

## Typed client ↔ server

Plain Nitro routes + typed `$fetch`/`useFetch` infer response types from route return types end-to-end;
shared Zod provides input validation + inferred input types. No tRPC (it would duplicate routing and fight
Better Auth's Nitro route mounting).

## Auth boundary

- Login is Better Auth only (`server/api/auth/[...all].ts`). Sessions read server-side via
  `auth.api.getSession({ headers })`, cached on `event.context`. Pages hydrate with `useSession(useFetch)`.
- Admin routes check `session.user.role === 'admin'`.
- **Calendar connections are separate from login** — see [calendar-sync.md](./calendar-sync.md). A user can
  log in with email/password and still connect Google/CalDAV/iCal calendars.

## Background work

A single Nitro scheduled task (`server/tasks/sync.ts`, croner engine) polls connected accounts on an interval
and runs incremental sync. The tested deployment target is **one persistent Node container**; on multi-replica
deployments, run migrations and the sync task as a dedicated one-shot/leader step to avoid races (see
[self-hosting.md](./self-hosting.md)).
