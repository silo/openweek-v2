# Self-Hosting

Self-hosting via Docker is a first-class use case. This doc is the operational contract: environment, secrets,
the compose stack, migrations, backups, and running behind a reverse proxy. (Scripts/files described here are
created in build Phase 1 — see [roadmap.md](./roadmap.md).)

## Deployment target

The tested target is **a single persistent Node container + PostgreSQL** via Docker Compose. The app must run
**persistently** (not serverless/scale-to-zero) because background calendar sync runs as an in-process Nitro
scheduled task. Multi-replica deployments are possible but require running migrations and sync as a dedicated
leader/one-shot step (see [Multi-replica](#multi-replica-notes)).

## Environment variables

| Var | Required | Example / format | Purpose |
|---|---|---|---|
| `DATABASE_URL` | yes | `postgres://user:pass@db:5432/openweek` | Postgres connection. |
| `BETTER_AUTH_SECRET` | yes | `openssl rand -base64 32` | Signs sessions. **Rotating it logs everyone out.** |
| `BETTER_AUTH_URL` | yes | `https://openweek.example.com` | Public origin; used for cookies + OAuth callbacks + `trustedOrigins`. |
| `OPENWEEK_ENCRYPTION_KEY` | yes | base64 of **32 bytes** (`openssl rand -base64 32`) | Encrypts CalDAV/Google calendar secrets. Distinct from `BETTER_AUTH_SECRET`. |
| `GOOGLE_CLIENT_ID` / `GOOGLE_CLIENT_SECRET` | optional | from Google Cloud Console | Google **login** and/or Google **calendar** connect. App runs fine without them (email/password + CalDAV/iCal still work). |
| `OPENWEEK_SYNC_INTERVAL` | optional | `10m` | Calendar poll interval. |
| `NODE_ENV` | yes | `production` | Enables secure cookies, etc. |

All env is **Zod-validated at boot** (`server/utils/runtime-config.ts`) so a misconfigured instance fails fast
with a clear message instead of crashing at runtime. A complete `.env.example` ships in the repo root.

## Secrets — generate them

```bash
# session secret
openssl rand -base64 32
# token-encryption key (must decode to exactly 32 bytes)
openssl rand -base64 32
```

Keep `BETTER_AUTH_SECRET` and `OPENWEEK_ENCRYPTION_KEY` **separate**. The Google account tokens used for
*login* are handled by Better Auth; the *calendar* secrets are encrypted by `OPENWEEK_ENCRYPTION_KEY`.

## Compose stack

`compose.yaml` defines:
- **`db`** — `postgres:16`, with a **named volume** for durability and a healthcheck.
- **`app`** — the Nuxt build, `depends_on: db (healthy)`, runs migrations on start, exposes port 3000.

```bash
cp .env.example .env        # then fill in secrets
docker compose up -d        # start db + app
docker compose logs -f app  # watch boot (env validation + migrations)
```

## Migrations

- We commit **SQL migrations** generated by `drizzle-kit generate`. Never use `push` in production.
- The app entrypoint runs `drizzle-kit migrate` on start (single-container case).
- Regenerate auth tables when an auth plugin/field changes:
  `pnpm auth:gen` (= `@better-auth/cli generate`) → `pnpm db:generate` → `pnpm db:migrate`.

## Backups & restore

Your tasks live in Postgres — back it up.

```bash
# backup
docker compose exec -T db pg_dump -U "$PGUSER" openweek > openweek-$(date +%F).sql
# restore
docker compose exec -T db psql -U "$PGUSER" -d openweek < openweek-YYYY-MM-DD.sql
```

Document a cron (host-side) for periodic dumps and verify restores. The named volume protects against
container loss but is **not** a backup.

## Reverse proxy (nginx / Traefik / Caddy)

Behind a proxy, auth breaks unless the public origin and forwarded headers are correct:

- Set `BETTER_AUTH_URL` to the **public** origin (https).
- Better Auth's `trustedOrigins` must include that origin; secure cookies are enabled when `NODE_ENV=production`.
- The proxy must forward `X-Forwarded-Proto` and `X-Forwarded-Host`, and Nitro must trust them.
- Google OAuth callback (if used): `${BETTER_AUTH_URL}/api/auth/callback/google` — add it in Google Cloud
  Console.

A minimal Caddy example:
```
openweek.example.com {
  reverse_proxy app:3000
}
```

## First-run

The **first user to register becomes `admin`**; everyone after is a normal user. Email verification is **off**
by default (a fresh self-host has no SMTP); password reset / verification are a later phase. Admin tools live
at `/admin/users`.

## Multi-replica notes

In-process scheduled sync and migrate-on-boot both assume a single instance. If you scale `app`:
- Run migrations as a separate **one-shot init service** (compose `depends_on`), not in every container.
- Run calendar sync from a single **leader** (or a dedicated worker), or two replicas will double-poll and hit
  provider rate limits.
